# Техническое описание проекта Creative Platform

## Роли

1. **Читатель**  
   Имеет возможность просматривать посты, комментировать их, ставить лайки/дизлайки, подписываться на публикаторов, отправлять жалобы на публикаторов. Может фильтровать посты в ленте по подпискам и категориям.

2. **Публикатор**  
   Имеет все возможности читателя, а также может публиковать посты с выбором категории.

3. **Администратор**  
   Имеет все возможности публикатора, а также может просматривать жалобы на посты, комментарии и пользователей, удалять посты и блокировать пользователей.

## Сущности

### 1. user (пользователь)
- `id` — первичный ключ, целочисленный, автоинкремент
- `name` — строка, not null
- `email` — строка, уникальный, nullable для OAuth-юзеров
- `github_id` — строка, уникальный, nullable
- `password` — строка, хранится хэш пароля, nullable для чистого OAuth
- `role_id` — целочисленный, внешний ключ на таблицу `role`, индексируемый
- `is_active` — логическое значение, по умолчанию `true`
- `created_at` — дата-время создания записи
- `updated_at` — дата-время обновления записи

### 2. role (роль)
- `id` — первичный ключ, целочисленный, автоинкремент
- `name` — строка, уникальное название роли: `reader`, `publisher`, `admin`
- `display_name` — строка, человекочитаемое название роли: Читатель, Публикатор, Администратор
- `description` — строка или текст, краткое описание назначения роли и её возможностей
- `is_active` — логическое значение, позволяет включать или отключать роль без удаления
- `created_at` — дата-время создания записи
- `updated_at` — дата-время обновления записи

### 3. post (пост)
- `id` — первичный ключ, целочисленный, автоинкремент
- `user_id` — целочисленный, внешний ключ на таблицу `user`, автор публикации, индексируемый
- `category_id` — целочисленный, внешний ключ на таблицу `category`, индексируемый
- `title` — строка, nullable, если публикация может быть без заголовка
- `content` — текст, nullable, текстовое содержимое публикации
- `content_type` — строка или enum: `text`, `image`, `audio`, `mixed`
- `media_path` — строка, nullable, путь к загруженному файлу или URL медиа
- `excerpt` — строка или текст, nullable, краткое описание публикации
- `status` — строка или enum: `draft`, `published`, `deleted`, по умолчанию `published`
- `published_at` — дата-время публикации, nullable для черновиков
- `created_at` — дата-время создания записи
- `updated_at` — дата-время обновления записи

### 4. comment (комментарий)
- `id` — первичный ключ, целочисленный, автоинкремент
- `post_id` — целочисленный, внешний ключ на таблицу `post`, индексируемый
- `user_id` — целочисленный, внешний ключ на таблицу `user`, автор комментария, индексируемый
- `parent_id` — целочисленный, nullable, внешний ключ на эту же таблицу `comment`; если вложенные ответы не нужны, поле можно убрать
- `content` — текст, not null
- `is_deleted` — логическое значение, по умолчанию `false`
- `created_at` — дата-время создания записи
- `updated_at` — дата-время обновления записи

### 5. subscribe (подписка)
- `id` — первичный ключ, целочисленный, автоинкремент
- `subscriber_id` — целочисленный, внешний ключ на таблицу `user`, кто подписался, индексируемый
- `publisher_id` — целочисленный, внешний ключ на таблицу `user`, на кого подписались, индексируемый
- `created_at` — дата-время создания записи
- `updated_at` — дата-время обновления записи

### 6. category (категория)
- `id` — первичный ключ, целочисленный, автоинкремент
- `name` — строка, уникальное название категории
- `slug` — строка, уникальный человекочитаемый идентификатор для URL и фильтрации
- `description` — строка или текст, nullable, описание категории
- `is_active` — логическое значение, по умолчанию `true`
- `created_at` — дата-время создания записи
- `updated_at` — дата-время обновления записи

### 7. reaction (реакция)
- `id` — первичный ключ, целочисленный, автоинкремент
- `post_id` — целочисленный, внешний ключ на таблицу `post`, индексируемый
- `user_id` — целочисленный, внешний ключ на таблицу `user`, кто поставил реакцию, индексируемый
- `type` — строка или enum: `like`, `dislike`
- `created_at` — дата-время создания записи
- `updated_at` — дата-время обновления записи

### 8. report (жалоба)
- `id` — первичный ключ, целочисленный, автоинкремент
- `user_id` — целочисленный, внешний ключ на таблицу `user`, кто отправил жалобу, индексируемый
- `target_type` — строка или enum: `post`, `comment`, `user`
- `target_id` — целочисленный, идентификатор сущности, на которую пожаловались
- `reason` — строка или enum: `spam`, `abuse`, `plagiarism`, `other`
- `message` — текст, nullable, комментарий пользователя к жалобе
- `status` — строка или enum: `pending`, `reviewed`, `rejected`, `accepted`, по умолчанию `pending`
- `reviewed_by` — целочисленный, nullable, внешний ключ на таблицу `user`, администратор, который обработал жалобу
- `reviewed_at` — дата-время, nullable
- `created_at` — дата-время создания записи
- `updated_at` — дата-время обновления записи

## Связи сущностей

| Связь | Тип | Как реализуется | Комментарий |
|---|---|---|---|
| `role -> user` | `1:N` | `users.role_id -> roles.id` | Одна роль может быть назначена многим пользователям, у пользователя одна роль |
| `user -> post` | `1:N` | `posts.user_id -> users.id` | Один пользователь может создать много постов, один пост имеет одного автора |
| `category -> post` | `1:N` | `posts.category_id -> categories.id` | Одна категория содержит много постов, один пост относится к одной категории |
| `post -> comment` | `1:N` | `comments.post_id -> posts.id` | У одного поста может быть много комментариев |
| `user -> comment` | `1:N` | `comments.user_id -> users.id` | Один пользователь может оставить много комментариев |
| `comment -> comment` | `1:N` | `comments.parent_id -> comments.id` | Нужно только если оставляешь ответы на комментарии; если нет — поле `parent_id` можно убрать |
| `user -> subscribe` как подписчик | `1:N` | `subscriptions.subscriber_id -> users.id` | Один пользователь может подписаться на многих публикаторов |
| `user -> subscribe` как публикатор | `1:N` | `subscriptions.publisher_id -> users.id` | На одного публикатора могут подписаться многие читатели |
| `user <-> user` через `subscribe` | `M:N` | связующая таблица `subscriptions` | Это самоссылочная many-to-many связь пользователей |
| `user -> reaction` | `1:N` | `reactions.user_id -> users.id` | Один пользователь может ставить много реакций на разные посты |
| `post -> reaction` | `1:N` | `reactions.post_id -> posts.id` | Один пост может иметь много реакций |
| `user <-> post` через `reaction` | `M:N` | связующая таблица `reactions` | Пользователь может реагировать на много постов, пост — получать реакции от многих пользователей |
| `user -> report` как автор жалобы | `1:N` | `reports.user_id -> users.id` | Один пользователь может отправить много жалоб |
| `user -> report` как модератор | `1:N` | `reports.reviewed_by -> users.id` | Один администратор может обработать много жалоб |
| `report -> target` | полиморфная | `target_type + target_id` | Жалоба относится либо к посту, либо к комментарию, либо к пользователю |

## Ограничения

| Объект | Ограничение | Тип ограничения | Зачем нужно |
|---|---|---|---|
| `users.email` | Уникальный email | `UNIQUE` | Один email не должен принадлежать нескольким пользователям |
| `users.github_id` | Уникальный GitHub ID, nullable | `UNIQUE` | Один GitHub-аккаунт должен быть привязан максимум к одному пользователю |
| `users.role_id` | Роль должна существовать в `roles` | `FOREIGN KEY` | Нельзя назначить пользователю несуществующую роль |
| `roles.name` | Уникальное системное имя роли | `UNIQUE` | Исключает дубли `reader`, `publisher`, `admin` |
| `categories.name` | Уникальное название категории | `UNIQUE` | Не даёт создать дублирующиеся категории |
| `categories.slug` | Уникальный slug | `UNIQUE` | Нужен для стабильных URL и фильтрации |
| `posts.user_id` | Автор поста должен существовать | `FOREIGN KEY` | Пост не может ссылаться на несуществующего пользователя |
| `posts.category_id` | Категория поста должна существовать | `FOREIGN KEY` | Пост обязан принадлежать валидной категории |
| `posts.status` | Только допустимые значения | `ENUM` или `CHECK` | Например: `draft`, `published`, `deleted` |
| `posts.content_type` | Только допустимые значения | `ENUM` или `CHECK` | Например: `text`, `image`, `audio`, `mixed` |
| `comments.post_id` | Комментируемый пост должен существовать | `FOREIGN KEY` | Нельзя создать комментарий без поста |
| `comments.user_id` | Автор комментария должен существовать | `FOREIGN KEY` | Нельзя создать комментарий без пользователя |
| `comments.parent_id` | Родительский комментарий должен существовать | `FOREIGN KEY` | Нужен только если есть вложенные ответы |
| `subscriptions.subscriber_id` | Подписчик должен существовать | `FOREIGN KEY` | Гарантирует корректную ссылку на пользователя |
| `subscriptions.publisher_id` | Публикатор должен существовать | `FOREIGN KEY` | Гарантирует корректную ссылку на пользователя |
| `subscriptions(subscriber_id, publisher_id)` | Одна подписка на одного автора только один раз | `UNIQUE` | Запрещает дубли подписок |
| `subscriptions` | Нельзя подписаться на самого себя | бизнес-правило / `CHECK` | Логически запрещённый сценарий |
| `reactions.post_id` | Пост должен существовать | `FOREIGN KEY` | Реакция всегда привязана к существующему посту |
| `reactions.user_id` | Пользователь должен существовать | `FOREIGN KEY` | Реакция должна иметь автора |
| `reactions(user_id, post_id)` | Только одна реакция пользователя на один пост | `UNIQUE` | Не даёт одновременно хранить несколько реакций одного пользователя на один пост |
| `reactions.type` | Только `like` или `dislike` | `ENUM` или `CHECK` | Ограничивает допустимые типы реакций |
| `reports.user_id` | Автор жалобы должен существовать | `FOREIGN KEY` | Жалоба должна иметь отправителя |
| `reports.reviewed_by` | Проверяющий админ должен существовать | `FOREIGN KEY` | Если жалоба обработана, обработчик должен быть валидным пользователем |
| `reports.target_type` | Только `post`, `comment`, `user` | `ENUM` или `CHECK` | Ограничивает тип объекта жалобы |
| `reports.status` | Только допустимые статусы | `ENUM` или `CHECK` | Например: `pending`, `reviewed`, `rejected`, `accepted` |

## Архитектура

Проект представляет собой локальное веб-приложение, состоящее из нескольких сервисов, запускаемых в `Docker Compose`: `nginx`, `laravel` (PHP-FPM), `fastapi` (Uvicorn), `mysql` и `redis`.

Все запросы пользователя сначала попадают в Nginx, который работает как единая точка входа и в зависимости от домена перенаправляет трафик либо в Laravel, либо в FastAPI.

Для локального запуска используются два виртуальных хоста, привязанных к `127.0.0.1`:
- `creative.localhost` — для Laravel
- `api.creative.localhost` — для FastAPI

Такой подход позволяет сохранить архитектурную идею ТЗ с разными доменами, но без деплоя на внешний сервер.

## Роли сервисов

### Nginx
Принимает HTTP/HTTPS-запросы от браузера, выполняет SSL-терминацию, делает редирект с HTTP на HTTPS и проксирует запросы внутрь Docker-сети на нужный контейнер. Для WebSocket Nginx отдельно пробрасывает `Upgrade` и `Connection`.

### Laravel
Является основным серверным приложением и отвечает за SSR на Blade, регистрацию и вход пользователей, OAuth через GitHub, создание и редактирование постов, комментарии, подписки, реакции и отправку жалоб. Laravel работает с MySQL на чтение и запись, а при значимых действиях публикует события в Redis Pub/Sub для realtime-слоя.

### FastAPI
Представляет отдельный сервис для REST API и WebSocket. Он читает данные из MySQL в режиме read-only, отдаёт JSON для списков постов, одного поста, фильтрации по категориям и публикациям конкретного автора, а также держит WebSocket-подключения клиентов и рассылает realtime-обновления.

### MySQL
Хранит основные данные приложения: пользователей, роли, посты, комментарии, подписки, категории, реакции и жалобы.

### Redis
Используется не как постоянное хранилище, а как промежуточный канал Pub/Sub между Laravel и FastAPI для доставки событий о новых публикациях и других значимых действиях.

## Поток запросов

При открытии основных страниц браузер отправляет запрос на `https://creative.localhost`, Nginx по заголовку `Host` направляет его в Laravel, а Laravel формирует HTML-страницу на сервере и возвращает её пользователю.

Когда фронтенду нужны API-данные, например список постов по категории или публикации конкретного автора, запрос отправляется на `https://api.creative.localhost`, после чего Nginx проксирует его в FastAPI. FastAPI читает данные из MySQL и возвращает JSON-ответ с корректными статус-кодами.

Для realtime-функционала React-компонент на странице открывает WebSocket-соединение с FastAPI через Nginx, например по адресу `wss://api.creative.localhost/ws`. Чтобы такое соединение работало через reverse proxy, в конфигурации Nginx обязательно должны быть включены проксирование WebSocket и передача заголовков `Upgrade` / `Connection`.

## Realtime-цепочка

Когда публикатор создаёт новый пост, браузер отправляет обычный HTTP-запрос в Laravel через `creative.localhost`. Laravel валидирует данные, сохраняет запись в MySQL, после чего публикует событие в Redis, например `post.created`, вместе с полезной нагрузкой: `post_id`, `author_id`, `category_id`, `title`, `content_type`, `published_at`.

FastAPI подписан на соответствующий Redis-канал и получает это событие без прямого HTTP-вызова от Laravel. После получения события FastAPI либо сразу рассылает его всем подключённым клиентам, либо фильтрует адресатов по подпискам, после чего отправляет нужным пользователям WebSocket-сообщение с данными о новой публикации.

На клиенте React-компонент принимает сообщение по WebSocket и обновляет интерфейс без перезагрузки страницы, например показывает уведомление о новой публикации публикатора или добавляет новый пост в ленту подписок.

## Пользовательские сценарии

1. **Регистрация и вход в систему**  
   Гость открывает страницу регистрации или входа на SSR-странице Laravel. Пользователь может зарегистрироваться по email и паролю либо войти через GitHub OAuth, после чего Laravel создаёт или находит запись в таблице `user`, проверяет роль и статус `is_active`, а затем открывает доступ к защищённым страницам.

2. **Просмотр ленты публикаций**  
   Читатель, публикатор или администратор открывает главную страницу приложения и получает серверно-сгенерированную ленту публикаций через Laravel.

3. **Фильтрация ленты по категориям**  
   Пользователь выбирает категорию в интерфейсе ленты, после чего система показывает только те записи, которые связаны с выбранной категорией через `post.category_id`.

4. **Фильтрация ленты по подпискам**  
   Читатель или публикатор выбирает режим просмотра публикаций только от тех авторов, на которых он подписан.

5. **Просмотр страницы отдельного поста**  
   Пользователь открывает страницу конкретной публикации и получает полную SSR-страницу поста, включая заголовок, текст, медиа, автора, категорию, реакции и комментарии.

6. **Создание поста публикатором**  
   Публикатор переходит на страницу создания поста, заполняет форму, указывает заголовок, текст, категорию и тип контента, а при необходимости прикрепляет изображение или аудио.

7. **Редактирование и удаление своего поста**  
   Публикатор открывает страницу редактирования только для собственной публикации.

8. **Комментирование поста**  
   Читатель, публикатор или администратор открывает страницу поста и оставляет комментарий к публикации.

9. **Ответ на комментарий**  
   Если в проекте используется `parent_id`, пользователь может ответить на существующий комментарий.

10. **Постановка лайка или дизлайка**  
    Авторизованный пользователь нажимает кнопку лайка или дизлайка на странице поста.

11. **Подписка на публикатора**  
    Читатель или публикатор открывает профиль автора или страницу его публикации и нажимает кнопку подписки.

12. **Получение уведомления о новой публикации**  
    Когда публикатор создаёт новый пост, Laravel публикует событие `post.created` в Redis, а FastAPI отправляет уведомление подписанным пользователям по WebSocket.

13. **Отправка жалобы**  
    Пользователь выбирает, на что подаётся жалоба: на пост, комментарий или пользователя.

14. **Просмотр жалоб администратором**  
    Администратор открывает защищённую страницу модерации и просматривает список жалоб.

15. **Обработка жалобы администратором**  
    Администратор принимает решение: отклонить жалобу, подтвердить её или выполнить модерационное действие.

16. **Удаление поста администратором**  
    Администратор удаляет публикацию через административный интерфейс.

17. **Блокировка пользователя администратором**  
    Администратор изменяет поле `is_active` пользователя на `false`.

18. **Получение данных через FastAPI**  
    Клиент отправляет запрос в FastAPI за списком постов, публикациями автора или одним постом.

19. **Подключение к WebSocket**  
    React-компонент устанавливает WebSocket-соединение с FastAPI через Nginx.

20. **Выход из системы**  
    Авторизованный пользователь завершает сессию и возвращается в гостевой режим.

## WebSocket-контракт

WebSocket-прокси через Nginx требует корректной передачи `Upgrade` и `Connection`, а для проекта важно заранее зафиксировать, какие именно события получают клиенты и как выглядит payload.

### Подключение
- Клиент открывает соединение: `wss://api.creative.localhost/ws?user_id=123`
- FastAPI регистрирует соединение пользователя и хранит его в менеджере подключений
- При отключении клиент удаляется из списка активных соединений

### События
- `post.created` — новая публикация автора
- `post.updated` — изменение публикации
- `post.deleted` — удаление публикации
- `comment.created` — новый комментарий на просматриваемом посте
- `report.created` — опционально, если нужен realtime в админке
